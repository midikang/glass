What is Glass?

Your invisible AI assistant that sees and listens to everything you do

The Product in 30 Seconds

Glass is a desktop app that lives on your computer like a silent assistant. It sees what's on your screen, listens to your audio (meetings, podcasts, conversations), and uses AI to help you understand and recall everything.

Think of it like having a photographic memory—but one that understands context. You can ask it questions like "What was that API endpoint mentioned in yesterday's standup?" and it knows the answer because it was listening.

👀

Sees Your Screen

Captures screenshots to understand what you're doing—like taking notes on what you're working on.

🎧

Listens to Audio

Records and transcribes meetings, videos, and conversations in real-time—like having a note-taker in every meeting.

🤖

AI-Powered Recall

Ask questions based on everything it's seen and heard—like searching your entire work history with natural language.

💡
Key Insight: Privacy by Design

Glass is built to be local-first. Your data stays on your machine unless you explicitly sign in to save it to the cloud. This is rare in AI apps—most send everything to their servers.

The Two Main Features

🎯
Ask: Context-Aware AI Chat

Like ChatGPT, but it sees your screen and knows what you've been doing. You can ask "What's the bug on line 42?" and it sees the code you're looking at.

🎙️
Listen: Meeting Intelligence

Records audio, transcribes it in real-time, generates summaries, and lets you search everything said. Like Otter.ai built into your desktop.

Both features work together. If you're in a meeting (Listen mode) and want to ask a question about what was just said, you can switch to Ask mode and it has full context from the transcript.

02

Meet the Main Actors

The key components that make Glass work

Think of Glass Like a Theater Production

Instead of classes and functions, think of the code as actors—each with a specific job. When you click "Listen" or "Ask," these actors work together like a choreographed performance.

Let's meet the cast:

⚡
Main Process (index.js)

The director—manages the whole show. Starts the app, creates windows, handles authentication, and coordinates all the other actors. Like a conductor keeping everyone in sync.

🪟
Window Manager

The stage designer—controls what windows appear on screen (Ask, Listen, Settings) and where they're positioned. Think of it like managing multiple screens in a presentation.

🎙️
Listen Service

The audio engineer—captures audio from your mic and system sound, coordinates transcription, and generates meeting summaries. It's the brains behind the "Listen" feature.

💬
Ask Service

The conversation manager—handles your AI chat, captures screenshots, sends everything to AI providers (OpenAI, Claude, etc.), and streams responses back to you.

🗄️
Database (SQLite or Firebase)

The memory vault—stores all your transcripts, chat history, session data, and settings. SQLite keeps it local; Firebase syncs to the cloud.

🌐
Web Frontend (Next.js)

The costume department—renders the UI you see. Built with Next.js and served inside the Electron app, so it looks like a website but runs locally.

Why This Architecture?

Glass uses Electron, which means it's a web app disguised as a desktop app. This gives you the best of both worlds:

🚀

Desktop Superpowers

Access to screen capture, audio recording, and local files—things websites can't do.

🎨

Modern UI

Built with React and Tailwind, so it's fast to develop and looks great.

🔌

Cross-Platform

Works on macOS, Windows, and Linux with mostly the same code.

💡
Pattern: Separation of Concerns

Each actor has one job. The Listen Service doesn't care about windows. The Window Manager doesn't care about audio. This makes the code easier to debug—if transcription breaks, you know exactly where to look.

How They Talk to Each Other

Actors can't just call each other directly (that would create a tangled mess). Instead, they communicate through bridges and events—like passing notes instead of shouting across a room.

CODE 
// From listenService.js
const internalBridge = require('../../bridge/internalBridge');

internalBridge.emit('window:requestVisibility', {
  name: 'listen',
  visible: true
});
PLAIN ENGLISH

Import the communication system that lets different parts of the app talk...

Send a message on the "window visibility" channel saying "hey, I need the Listen window to appear"...

Specify which window (the Listen window) and what to do with it (make it visible).

The Window Manager is listening for that message. When it arrives, it shows the Listen window. Neither service needs to know the other exists—they just send and receive messages through the bridge.

The Code Behind the Magic

When you click "Listen," here's what the code does:

CODE 
// From listenService.js - initializeSession()
async initializeSession() {
  this.currentSessionId = await sessionRepository.create('listen');
  
  await this.sttService.startTranscription(this.currentSessionId);
  await this.summaryService.initialize(this.currentSessionId);
}
PLAIN ENGLISH

Create a new session in the database and remember its ID (like starting a new notebook and labeling it)...

Tell the speech-to-text service to start listening and transcribing audio for this session...

Initialize the summary service so it can generate meeting notes when we're done.

Notice how clean this is. The Listen Service doesn't know how transcription works—it just says "start transcribing." The STT Service handles all the messy audio processing details.

💡
Pattern: Delegation

Each service delegates to specialists. The Listen Service doesn't process audio itself—it delegates to STT Service. This is like a project manager who assigns tasks to experts instead of doing everything themselves. If transcription breaks, you debug STT Service, not the whole Listen flow.

Where Does the Data Live?

Glass supports two storage modes:

💾

Local (SQLite)

Data stays on your computer in a SQLite database file. Fast, private, works offline. No account needed.

☁️

Cloud (Firebase)

Data syncs to Firebase when you sign in. Syncs across devices, backed up automatically. Requires a Pickle account.

The clever part? The code doesn't care which one you use. There's an adapter pattern that swaps between SQLite and Firebase without changing the rest of the app.

CODE 
// From askRepository (repositories/index.js)
const currentStorage = authService.getStorageMode();

if (currentStorage === 'firebase') {
  return firebaseRepo.getAllAiMessagesBySessionId(sessionId);
} else {
  return sqliteRepo.getAllAiMessagesBySessionId(sessionId);
}
PLAIN ENGLISH

Check which storage mode the user is in (local or cloud)...

If they're using Firebase (cloud), fetch data from Firebase...

Otherwise, fetch from the local SQLite database. Either way, return the same data format.

This means you can switch from local to cloud (or vice versa) without rewriting any code. The Ask Service doesn't care where data comes from—it just asks the repository and gets an answer.

04

AI Providers & External Services

How Glass talks to OpenAI, Claude, Gemini, and more

The AI Provider Zoo

Glass doesn't have its own AI model. Instead, it connects to external AI services—like how a calculator app doesn't do math itself, it uses the phone's processor.

Here are all the AI providers Glass supports:

🤖
OpenAI (GPT-4, GPT-4o)

For chat responses. The default for most users.

🧠
Anthropic Claude

Alternative to OpenAI. Known for longer context windows (remembers more history).

✨
Google Gemini

Google's AI model. Good for vision tasks (analyzing screenshots).

🦙
Ollama (Local AI)

Runs AI models on your own computer—no API key needed, completely private.

🎙️
Deepgram & Whisper (Speech-to-Text)

Converts audio to text. Deepgram is cloud-based; Whisper runs locally.

The Factory Pattern: Swapping Providers

Glass uses a factory to create the right AI provider based on your settings. Like ordering from a restaurant menu—you say "burger" and the kitchen figures out how to make it.

CODE 
// From ai/factory.js - createStreamingLLM()
function createStreamingLLM(provider) {
  switch (provider) {
    case 'openai':
      return new OpenAIProvider();
    case 'anthropic':
      return new AnthropicProvider();
    case 'gemini':
      return new GeminiProvider();
    case 'ollama':
      return new OllamaProvider();
  }
}
PLAIN ENGLISH

Define a function that creates the right AI provider based on a name...

If the user wants OpenAI, create an OpenAI connection...

If they want Claude (Anthropic), create that instead...

Same for Gemini and Ollama. Return whichever provider was requested.

Now the Ask Service can just call createStreamingLLM('openai') and not care about API keys, URL endpoints, or request formatting—the factory handles all that.

💡
Pattern: Abstraction

The Ask Service doesn't know how to talk to OpenAI vs Claude—it just knows "send this message, get a response." Each provider handles the messy details (API keys, rate limits, error formats). This means adding a new AI provider (say, Perplexity) only requires writing one new file, not changing the entire app.

Streaming Responses

When you ask Glass a question, you don't wait 30 seconds for a full response—you see words appear one by one, like typing. This is called streaming.

Here's how it works:

1

Send question to AI
Glass sends your message + context to OpenAI

→
2

AI streams back chunks
OpenAI sends "The..." then "answer..." then "is..."

→
3

Display as they arrive
Each chunk gets appended to your screen immediately

CODE 
// From openai.js - stream()
const stream = await openai.chat.completions.create({
  model: 'gpt-4o',
  messages: messages,
  stream: true
});

for await (const chunk of stream) {
  const text = chunk.choices[0].delta.content || '';
  onChunk(text);
}
PLAIN ENGLISH

Ask OpenAI for a response, but tell it to stream results back instead of waiting for the full answer...

Loop through each chunk that arrives (like checking your mailbox repeatedly for new letters)...

Extract the text from each chunk and immediately send it to the UI to display.

This makes the app feel instant. Instead of a loading spinner, users see progress immediately.

Clever Engineering Patterns

The tricks that make Glass fast, reliable, and maintainable

Patterns You Can Steal

These are the engineering decisions that make Glass work well—and they're patterns you can request from AI when building your own apps.

Pattern 1: Repository Pattern (Swap Storage Backends)

Instead of scattering database calls everywhere, Glass uses repositories—a single place for all data operations for each feature.

Think of it like a librarian. You don't walk into the library and start pulling books off shelves yourself. You ask the librarian, and they know exactly where everything is.

CODE 
// From ask/repositories/index.js
const sqliteRepo = require('./sqlite.repository');
const firebaseRepo = require('./firebase.repository');
const authService = require('../../common/services/authService');

module.exports = {
  getAllAiMessagesBySessionId(sessionId) {
    const mode = authService.getStorageMode();
    return mode === 'firebase'
      ? firebaseRepo.getAllAiMessagesBySessionId(sessionId)
      : sqliteRepo.getAllAiMessagesBySessionId(sessionId);
  }
};
PLAIN ENGLISH

Load both the SQLite and Firebase implementations...

Check which storage mode the user is in...

Route the request to the right backend. The caller doesn't know or care which one is used—they just get data back.

🎯
When to Use This

Use the repository pattern when you might switch databases later (SQLite → Postgres, Firebase → Supabase). It lets you swap the backend without rewriting your app. When talking to AI: "Create a repository pattern for user data that supports both SQLite and Firebase."

Pattern 2: Event Bridge (Decouple Components)

Services communicate via an event bridge—a message bus where actors send and listen for events. Like a radio station: you broadcast a message, and anyone tuned to that channel hears it.

This prevents tight coupling. The Listen Service doesn't call the Window Manager directly—it just emits "I need the window visible" and the Window Manager picks it up.

✅

With Event Bridge

Listen Service emits an event. Window Manager listens for it. They don't know about each other—easy to test and modify independently.

❌

Without Event Bridge

Listen Service imports Window Manager and calls it directly. Now they're tangled—changing one might break the other.

🎯
When to Use This

Use events when multiple parts of your app need to react to the same action. For example, when a user logs in, you might need to update the UI, fetch data, and log analytics—all at once. When talking to AI: "Set up an event emitter to broadcast user login events."

Pattern 3: Service Layer (Business Logic Isolation)

The service layer is where all the "how do we actually do this?" logic lives. The UI doesn't process audio—it asks the Listen Service. The Listen Service doesn't know about buttons—it just exposes functions like startRecording().

Think of it like a restaurant. The waiter (UI) takes your order and delivers food, but they don't cook. The kitchen (service layer) handles all the cooking. The waiter doesn't need to know how to make pasta—they just need to know how to order it.

CODE 
// UI calls this simple function:
await listenService.initializeSession();

// Behind the scenes, the service does:
async initializeSession() {
  this.currentSessionId = await sessionRepo.create('listen');
  await this.sttService.startTranscription(this.currentSessionId);
  await this.summaryService.initialize(this.currentSessionId);
}
PLAIN ENGLISH

The UI just calls one function—it doesn't care about the steps...

The service handles all the orchestration: creating a session, starting transcription, initializing summaries...

If the UI needs to change (new button layout, mobile version), it doesn't affect the logic—it's all in the service.

🎯
When to Use This

Always. Seriously—put your logic in services, not in UI components. UI should only handle display and user input. When talking to AI: "Create a service layer for handling payments that the UI can call with simple functions."

Pattern 4: Graceful Shutdown (Cleanup on Exit)

When you quit Glass, it doesn't just close—it cleans up first. Audio stops recording, sessions get marked as ended, the database closes properly. This prevents corrupted data.

It's like leaving a campsite. You don't just walk away—you put out the fire, pack your tent, and leave no trace.

CODE 
// From index.js - before-quit event
app.on('before-quit', async (event) => {
  event.preventDefault();
  
  await listenService.closeSession();
  await sessionRepository.endAllActiveSessions();
  await ollamaService.shutdown();
  databaseInitializer.close();
  
  app.exit(0);
});
PLAIN ENGLISH

When the app tries to quit, stop it temporarily...

Close any active recording session (stop audio capture)...

Mark all sessions in the database as ended...

Shut down the local AI service if running...

Close the database connection cleanly, then exit.

🎯
When to Use This

Whenever your app writes to files or databases. If the user force-quits your app mid-save, you could lose data. Graceful shutdown ensures cleanup happens first. When talking to AI: "Add a graceful shutdown handler that saves state before the app closes."

The Complete Architecture

How all the pieces fit together

The Full System Diagram

Let's zoom out and see the entire architecture. Here's how Glass is structured from top to bottom:

🖥️
Desktop Layer (Electron)

The shell that runs everything—provides window management, screen capture, audio access, and local file system.

⚙️
Main Process (Node.js)

The backend that coordinates all services—Listen, Ask, Settings, Auth, Database. Runs in the background.

🌐
Web Frontend (Next.js)

The UI you see—built with React and Tailwind, served as a local website inside Electron.

🗄️
Storage Layer (SQLite / Firebase)

Where all data is stored—transcripts, chat history, sessions, settings. Swappable between local and cloud.

☁️
External Services

AI providers (OpenAI, Claude), speech-to-text (Deepgram), authentication (Firebase Auth). Glass talks to these via APIs.

Why This Architecture Works

🔧

Easy to Debug

If transcription fails, you know it's in the STT Service. If AI responses are slow, check the provider layer. Isolation makes bugs predictable.

🚀

Easy to Extend

Want to add a new AI provider? Write one file. Want to support Postgres? Write one repository. The rest of the app doesn't change.

🧪

Easy to Test

Each service can be tested independently. You can test the Ask Service without opening windows. You can test the database without running the whole app.

💡
The Meta-Pattern: Modularity

All the patterns we've seen (repository, service layer, event bridge, factory) have one thing in common: modularity. Each piece is independent. This is the single most important principle in software—keep things separate, and they're easier to understand, change, and fix.
